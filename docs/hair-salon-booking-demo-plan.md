# 理发店预约运营中枢前端 Demo 规划

## 1. 产品定位

这个 Demo 的核心定位不是一个简单的“在线约时间”工具，而是一套面向理发店、美发沙龙和连锁门店的门店运营中枢。它需要把顾客预约、发型师排班、到店服务、收银结算、会员营销、员工绩效、库存消耗和经营报表串成一条完整链路。

这套系统后续可以作为可销售给不同理发店的前端产品模板或 SaaS 方案。不同门店可以定制 logo、门店名称、主题色、服务项目、员工资料和营销内容。当前 Demo 默认品牌名为“发哥沙龙”。

第一版是纯前端展示 Demo，技术栈采用 React + Vite。所有数据均使用本地 mock，不接后台接口，不做真实支付，不接真实微信推送，也不实现真实权限鉴权。

展示顺序按三端组织：

1. Admin 门店后台
2. 员工端
3. 会员/顾客端

目标是让演示对象一眼看出：这不是一个预约小工具，而是一个能帮老板管人、管钱、管客、管货、管服务质量的经营系统。

## 2. 现有 9 大功能模块 Review

### 2.1 顾客端：让客人能自己约

现有设想完整覆盖了顾客预约前链路，包括门店展示、服务项目、发型师展示、在线预约、预约规则、排队取号和微信提醒。

Demo 中建议重点呈现：

- 门店首页：地址、电话、营业时间、环境图、当前排队状态。
- 服务项目：洗剪吹、染发、烫发、护理、接发等，展示图片、时长、价格、热度。
- 发型师卡片：头像、擅长风格、评分、预约数、可约时间。
- 预约流程：选项目、选发型师、选日期时段、填写手机号、确认预约。
- 排队取号：展示前方人数、预计等待时间、当前叫号。

微信提醒在前端 Demo 中用消息卡片模拟即可，不需要真实推送。

### 2.2 预约核心：别让发型师撞单

这一块是系统可信度的关键。预约系统要展示“资源占用”的概念，而不是只展示日历。

Demo 中建议重点呈现：

- 时间格：按 15 或 30 分钟切分。
- 服务时长：女士烫染等长服务连续占用多个时间格。
- 资源占用：发型师、工位、服务项目、占用时长同时展示。
- 防超卖：已占用时段显示为不可选。
- 改期取消：展示预约状态和规则提示。
- 临时停约：模拟发型师请假后关闭对应时段。

### 2.3 门店后台：老板每天看这一套

后台是第一优先级页面，应该做成完整的门店经营工作台。

Demo 中建议重点呈现：

- 今日看板：预约总数、待到店、服务中、已完成、爽约、今日实收。
- 预约时间轴：按时间展示顾客、项目、发型师、状态。
- 手动改单：用侧边详情面板模拟换项目、换发型师、改状态。
- 直接到店：散客开单入口。
- 派单抢单：展示“指定发型师”和“待接单”状态。
- 权限管理：第一版不做真实鉴权，只在 UI 上展示老板、店长、前台、发型师、收银员的角色视角。

### 2.4 会员营销：把散客变回头客

会员模块是提升商业价值的重要部分，应避免只做余额卡。

Demo 中建议重点呈现：

- 电子会员卡：余额、次卡、套餐、积分、等级。
- 会员档案：手机号、性别、生日、常做项目、指定发型师。
- 消费记录：项目、金额、服务人员、支付方式。
- 标签分组：染烫客、高消费客、久未到店客、新客、生日月客户。
- 营销工具：优惠券、体验价、老带新、充值送、生日礼。
- 复购提醒：染烫后 6-8 周提醒补色。

### 2.5 收银结算：钱要对得上

收银模块要展示“预约到店后一键转服务单，再结算，再自动拆提成”的闭环。

Demo 中建议重点呈现：

- 多方式收款：微信、支付宝、现金、储值卡、次卡、团购券。
- 卡项核销：次卡、套餐、期卡扣减。
- 聚合核销：模拟美团、抖音、小红书团购码统一核销。
- 挂账记账：熟客或企业客户先挂账。
- 退款处理：原路退、部分退款、退款记录。
- 对账报表：日流水、实收、应收、退款、平台订单。

### 2.6 员工绩效：提成别靠手工算

员工绩效是老板和员工都会关心的卖点，应做成透明可解释的模块。

Demo 中建议重点呈现：

- 业绩归属：服务业绩、售卡业绩、产品销售分别归属。
- 提成规则：项目提成、售卡提成、产品提成、充值提成。
- 考勤打卡：上下班、请假、休息日同步排班。
- 技师端：今日排班、预约、待服务、已完成。
- 工资核算：服务单数、销售额、办卡金额、预计提成。

### 2.7 进销存：烫染药水别乱丢

库存模块可以作为高级卖点，不需要第一版做复杂进销存流程，但要让老板看到成本控制能力。

Demo 中建议重点呈现：

- 商品管理：洗发水、染膏、烫发水、护发素。
- 库存预警：低库存、即将过期、建议补货。
- 领用扣减：烫染服务完成后自动扣减耗材。
- 采购入库：模拟进货、退货、盘点记录。
- 成本利润：服务收入、耗材成本、毛利估算。

### 2.8 评价与客户维护：服务留痕

客户维护模块要和会员档案、员工绩效、复购提醒联动。

Demo 中建议重点呈现：

- 服务评价：发型师、环境、等待时间评分。
- 投诉售后：登记不满意原因、安排返工或退款。
- 回访任务：染烫后几天主动回访效果。
- 流失提醒：老客超过 60 天未到店自动提示跟进。
- 图片档案：做发前后图片、配方备注、风格标签。

### 2.9 数据报表：老板靠它做决策

报表模块是 Admin 后台的收束点，应服务于经营判断，而不是堆图表。

Demo 中建议重点呈现：

- 客流分析：每日客流、高峰时段、预约与到店对比。
- 项目热度：洗剪吹、染发、烫发、护理等占比。
- 员工排行：接单数、客单价、好评率、退款率。
- 会员分析：新老客占比、复购率、留存情况。
- 营收报表：日/月/年营业额、定金、退款、实收。
- 导出能力：预约名单、财务流水、提成表导出 Excel 的按钮模拟。

## 3. 可增强的 Fancy 卖点

### 3.1 AI 智能排班与防撞单

根据服务项目时长、发型师技能、工位占用、历史高峰、员工请假自动推荐最优预约时间。前端 Demo 中可展示“AI 推荐”标签和推荐理由，例如“该时段空闲工位充足，预计等待 0 分钟”。

### 3.2 客户画像与复购雷达

为会员生成画像卡片，包括消费频次、偏好项目、指定发型师、客单价、流失风险、下一步营销建议。比如“染烫高价值客，距离上次补色 52 天，建议发送补色券”。

### 3.3 发型档案时间线

每个顾客有做发历史，包括做发前照片、做发后照片、染膏配方、烫发药水、护理建议和发型师备注。这个功能很适合美发行业，卖点比普通会员档案更强。

### 3.4 经营大屏

做一个高视觉冲击的 Admin 首屏，展示今日预约、实时客流、收入、员工排行、服务热度、库存预警、会员转化。适合给老板或投资人演示。

### 3.5 智能收银单

预约到店后一键转服务单，服务完成后自动生成结算单，自动识别可用会员卡、次卡、团购券和优惠券，并展示提成拆分。

### 3.6 营销自动化

支持生日礼、久未到店召回、染烫补色提醒、老带新奖励、充值送活动。Demo 中可以用“自动化任务队列”展示系统正在帮门店运营客户。

### 3.7 员工绩效透明化

员工端展示今日服务、已完成单数、预计提成、客户评价和待回访任务，让员工知道自己为什么拿这些钱，也减少月底人工算账争议。

### 3.8 动态排队与等候体验

顾客到店前可以看到前方人数、预计等待时间、当前服务进度。门店端则能看到排队池、迟到风险和临时加塞提醒。

### 3.9 产品与耗材成本联动

烫染护理项目完成后自动扣减染膏、药水、护理产品库存，并粗略计算服务毛利。这个点能把预约系统升级成经营系统。

## 4. 前端 Demo 范围

第一版只做纯前端 mock，范围如下：

- 新建 React + Vite 项目。
- 使用本地 mock 数据驱动所有页面。
- 不接真实后台 API。
- 不做真实登录、注册、鉴权。
- 不接真实微信、短信、支付、扫码、Excel 导出。
- 不做真实拖拽排班算法，但可以用 UI 模拟“拖拽改单”和“AI 推荐排班”。
- 所有交互只更新前端 state，刷新后恢复初始 mock 数据。

优先展示完整产品感和高级视觉，不追求业务规则的全部真实落地。

## 5. 三端页面规划

### 5.1 Admin 门店后台

Admin 是第一展示重点，面向老板、店长、前台、收银员。

建议页面：

- 经营看板：首页大屏，展示今日预约、客流、营收、会员增长、员工排行、库存预警。
- 预约排班：时间轴、发型师列、工位占用、预约状态、临时停约。
- 收银结算：服务单、支付方式、会员卡/次卡/团购券核销、退款记录、提成拆分。
- 会员中心：会员列表、标签筛选、消费记录、客户画像、复购提醒。
- 员工绩效：排班、服务单数、销售额、提成、评价。
- 库存耗材：商品库存、预警、领用记录、毛利估算。
- 数据报表：营收、客流、项目热度、员工排行、会员复购。

### 5.2 员工端

员工端建议按移动端尺寸设计，面向发型师和技师。

建议页面：

- 今日工作台：上班状态、今日预约、服务中、待服务、已完成。
- 我的预约：顾客信息、项目、备注、会员偏好、历史发型档案。
- 服务详情：服务步骤、耗材记录、完成服务、添加备注。
- 我的绩效：今日业绩、预计提成、好评率、售卡/产品销售。
- 回访任务：染烫后回访、投诉售后、流失客户跟进。

### 5.3 会员/顾客端

会员端建议按手机体验设计，面向消费者。

建议页面：

- 门店首页：门店环境、地址、营业时间、排队状态。
- 服务项目：项目图片、价格、时长、推荐标签。
- 发型师选择：头像、评分、擅长风格、预约数、可约状态。
- 在线预约：项目、发型师、日期时段、手机号、备注。
- 预约成功：预约详情、到店提醒、取消/改期入口。
- 排队取号：当前叫号、前方人数、预计等待时间。
- 会员卡：余额、次卡、积分、优惠券、消费记录。

## 6. Mock 数据设计

建议将 mock 数据按业务域拆分，便于后续接真实接口。

核心数据：

- Store：门店名称、地址、电话、营业时间、环境图、当前排队信息。
- ServiceItem：服务名称、分类、图片、价格、时长、耗材成本、推荐标签。
- Stylist：姓名、头像、职位、擅长、评分、预约数、今日排班、可约状态。
- Appointment：顾客、项目、发型师、工位、开始时间、结束时间、状态、备注。
- Member：姓名、手机号、等级、余额、次卡、积分、标签、生日、指定发型师。
- Order：服务单、支付方式、优惠、退款、实收、应收、提成拆分。
- InventoryItem：商品名、分类、库存、单位、预警线、成本、最近领用。
- Performance：员工、服务单数、销售额、售卡金额、提成、评分。
- ReviewTask：评价、投诉、回访任务、流失提醒、处理状态。

状态枚举建议：

- 预约状态：待到店、服务中、已完成、已取消、爽约。
- 支付方式：微信、支付宝、现金、储值卡、次卡、团购券、挂账。
- 员工状态：上班中、休息、请假、忙碌、可接单。
- 任务状态：待处理、处理中、已完成。

## 7. 视觉与交互方向

整体视觉应体现“高级门店运营系统”，而不是普通表单后台。

设计建议：

- 当前 Demo 使用“发哥沙龙”作为默认品牌露出，后续品牌名、logo、主题色、门店图片都应抽象为可替换配置。
- Admin 使用深色高级 SaaS 风格，搭配清晰的数据色和高对比状态标签。
- 员工端使用移动优先布局，强调今日任务、下一位顾客和预计提成。
- 会员端使用精品沙龙风格，重点展示服务图片、发型师卡片和顺滑预约流程。
- 使用图标按钮、状态标签、时间轴、热力格、指标卡、头像组和侧边详情面板提升质感。
- 避免页面只由卡片堆叠构成，重点页面应有清晰的信息密度和业务动线。
- 不做纯营销落地页，打开即进入可操作 Demo。
- 文字和按钮在桌面与移动尺寸下都不能溢出或互相遮挡。

建议首屏重点：

- 左侧：模块导航和门店状态。
- 中间：今日预约时间轴与实时经营数据。
- 右侧：AI 排班建议、库存预警、会员复购提醒、员工排行。

## 8. 验收与测试标准

功能验收：

- 可以在 Admin、员工端、会员端之间切换。
- Admin 可查看今日看板、预约排班、收银、会员、绩效、库存、报表。
- 员工端可查看今日排班、服务详情、预计提成、回访任务。
- 会员端可浏览门店、选择服务、选择发型师、选择时段、生成预约预览。
- 预约状态切换、支付方式选择、会员详情查看等交互均能在前端 state 中体现。

视觉验收：

- 桌面端 Admin 页面信息密度高，但不拥挤。
- 移动端员工端和会员端可读、可点、无横向溢出。
- 关键状态颜色清晰区分：待到店、服务中、已完成、爽约、低库存、待回访。
- 页面有足够视觉卖点，适合演示给老板、客户或投资人。

技术验收：

- `npm install` 可安装依赖。
- `npm run dev` 可启动本地开发服务器。
- `npm run build` 可成功构建。
- 项目不依赖任何真实后端环境变量。
- 刷新页面后恢复初始 mock 数据是可接受行为。

## 9. 默认假设

- 技术栈固定为 React + Vite。
- 当前 Demo 默认门店品牌为“发哥沙龙”。
- 系统后续面向多家理发店销售，应支持替换 logo、品牌名称、主题色和门店素材。
- 第一版以演示效果为主，不实现复杂业务算法。
- 所有接口、支付、扫码、微信提醒、Excel 导出都用前端 UI 模拟。
- 不实现真实多角色权限，只通过入口和页面状态展示不同角色视角。
- 服务图片可先使用高质量远程占位图或本地 mock 图，后续再替换为品牌素材。
- 第一版展示顺序固定为 Admin 门店后台、员工端、会员/顾客端。
- 后续如果进入真实产品阶段，再补充接口协议、权限模型、数据库结构、支付/推送/导出能力。
